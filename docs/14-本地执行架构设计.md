## 设计目标

DoraMate 的“本地执行”不是指在浏览器中直接运行节点，而是指：

- 浏览器负责可视化编排和运行状态展示
- 本地代理负责文件系统、进程控制和原生对话框
- 真正的数据流执行仍由本机 DORA runtime 承担

当前实现的目标有三个：

1. 让用户在前端中完成数据流编排、保存、运行和停止
2. 把浏览器做不到的本地能力收敛到 `127.0.0.1:52100` 的 LocalAgent
3. 在本地执行过程中提供状态反馈、日志反馈和错误诊断

## 真实架构边界

当前仓库中的本地执行能力由两个项目协作完成：

- `doramate-frontend`
- `doramate-localagent`

### `doramate-frontend` 的角色

前端基于 Leptos + WebAssembly，实现：

- 画布编辑
- 节点与连线管理
- DoraMate 内部数据模型维护
- DoraMate YAML / DORA YAML 转换
- 运行与停止按钮
- 状态面板与日志面板
- 最近文件、模板管理、快捷键等交互

前端不会直接：

- 拉起本地节点进程
- 调用 `dora start`
- 打开原生目录选择器
- 直接读写宿主机文件系统

### `doramate-localagent` 的角色

LocalAgent 基于 Axum + Tokio，实现：

- 本地 HTTP API
- WebSocket 状态流和日志流
- 工作目录与 YAML 文件落盘
- `dora` CLI 检测与调用
- DORA coordinator / daemon 自启动
- 原生文件选择与保存对话框
- 节点模板配置读写

当前 LocalAgent 的核心逻辑集中在 [`main.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-localagent/src/main.rs) 中，文件规模约 3103 行。

### 真正执行仍由 DORA 负责

当前运行按钮最终走到的是：

```bash
dora start --detach ...
```

停止按钮最终走到的是：

```bash
dora stop ...
```

因此 DoraMate 当前的本地执行架构，本质是“前端编排 + 本地代理桥接 + DORA runtime 执行”。

## 整体链路

```text
doramate-frontend
  ├─ 编辑 Dataflow
  ├─ 生成 YAML
  ├─ 调用 LocalAgent HTTP API
  └─ 订阅状态与日志 WebSocket
           │
           ▼
doramate-localagent (127.0.0.1:52100)
  ├─ /api/run
  ├─ /api/stop
  ├─ /api/health
  ├─ /api/status/:process_id
  ├─ /api/status-stream/:process_id
  ├─ /api/logs/:process_id
  ├─ /api/select-directory
  ├─ /api/open-dataflow-file
  ├─ /api/save-dataflow-file
  └─ /api/node-templates-config
           │
           ▼
本机 DORA CLI
  ├─ dora coordinator
  ├─ dora daemon
  ├─ dora start
  └─ dora stop
           │
           ▼
本地工作目录 / YAML 文件 / 节点进程
```

这个结构有两个关键特点：

- 浏览器边界和系统边界划分清晰
- 所有本地能力统一收敛到 LocalAgent，而不是分散在前端中绕过浏览器限制

## 前端职责

### 1. 维护 DoraMate 内部数据模型

前端核心类型定义在 [`types.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/types.rs)：

- `Dataflow`: 画布内部格式
- `Node`
- `Connection`
- `DoraDataflow`: 运行时导出格式

其中 `DoraDataflow` 支持 `__doramate__` 元字段，用于保存布局和可视化信息。这意味着 DoraMate 保存的 YAML 和真正交给 DORA 的运行 YAML 并不完全相同。

### 2. 运行前先完成 YAML 转换

点击运行按钮后，前端会：

1. 从当前画布读取 `Dataflow`
2. 调用 `converter::dataflow_to_yaml`
3. 获取当前工作目录
4. 调用 [`api::run_dataflow`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/utils/api.rs)

这说明前端是“内容生成者”，不是“进程执行者”。

### 3. 运行成功后驱动 UI 进入运行态

前端运行成功后会做这些事：

- 保存 `process_id`
- 标记 `is_running = true`
- 将节点状态初始化为 `Starting`
- 打开日志面板
- 连接状态流 WebSocket
- 启动 HTTP 状态轮询
- 启动运行时长计时器

这部分逻辑主要位于 [`lib.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/lib.rs)。

### 4. 状态流和轮询同时存在

当前前端并不是只依赖一种方式，而是同时使用：

- `StatusWebSocket` 连接 `/api/status-stream/:process_id`
- 每 2 秒调用一次 `/api/status/:process_id`

这样做的目的很明确：

- WebSocket 提供低延迟更新
- 轮询在状态流异常时兜底

对当前阶段来说，这是比“只做一种”更稳妥的实现。

### 5. 日志面板已经具备基础可观测性

[`log_panel.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/components/log_panel.rs) 当前支持：

- WebSocket 连接状态提示
- 自动滚动开关
- 错误 toast
- 错误角标
- 按日志级别过滤
- 按节点 ID 过滤
- 关键字搜索
- 导出日志

这意味着 DoraMate 当前不仅能发起本地执行，也已经具备基本的运行观测界面。

## LocalAgent 职责

当前 LocalAgent 提供的接口面远不止 `run / stop / health`。

### 1. 当前接口列表

| 路由 | 方法 | 作用 |
|------|------|------|
| `/api/health` | `GET` | 返回 LocalAgent 与 DORA runtime 状态 |
| `/api/run` | `POST` | 启动数据流 |
| `/api/stop` | `POST` | 停止数据流 |
| `/api/select-directory` | `POST` | 打开原生目录选择器 |
| `/api/open-dataflow-file` | `POST` | 打开文件并读取 YAML |
| `/api/read-dataflow-file` | `POST` | 按路径读 YAML |
| `/api/save-dataflow-file` | `POST` | 打开保存对话框并写 YAML |
| `/api/write-dataflow-file` | `POST` | 按路径写 YAML |
| `/api/node-templates-config` | `GET/POST` | 读写模板配置 |
| `/api/status/:process_id` | `GET` | 返回状态快照 |
| `/api/status-stream/:process_id` | `GET` + WS | 推送状态流 |
| `/api/logs/:process_id` | `GET` + WS | 推送日志流 |

### 2. 当前状态存储模型

LocalAgent 使用：

- `AppState`
- `Arc<Mutex<HashMap<String, DoraProcess>>>`

当前没有数据库，也没有外部状态服务。运行中的流程信息都保存在内存注册表中。

`DoraProcess` 当前包含：

- `yaml_path`
- `started_at`
- `dataflow_uuid`
- `log_tx`
- `log_backlog`

这些字段分别支撑：

- 停止流程
- 计算运行时长
- 推送日志
- 回放 backlog
- 状态查询时重新解析 YAML

### 3. 健康检查比前端当前消费的更丰富

后端 `HealthResponse` 当前返回：

- `status`
- `version`
- `dora_installed`
- `dora_coordinator_running`
- `dora_daemon_running`

但前端 `api.rs` 当前只声明了：

- `status`
- `version`
- `dora_installed`

这说明后端已具备更丰富的本地环境诊断能力，前端尚未完全消费这些字段。

## 运行链路

`/api/run` 是当前架构最核心的能力。

### 1. 输入是 YAML 文本，不是文件路径

前端提交的数据结构为：

```json
{
  "dataflow_yaml": "...",
  "working_dir": "..."
}
```

因此本地执行的职责划分是：

- 前端负责生成 YAML 内容
- LocalAgent 负责决定文件如何落盘、进程如何启动

### 2. LocalAgent 会清洗 DoraMate 元数据

如果 YAML 中带有 `__doramate__` 元字段，LocalAgent 会先调用 `extract_clean_dora_yaml`，移除可视化元数据，只保留 DORA 真正可执行的部分。

### 3. 运行文件写入工作目录

当前实现会写入：

```text
<working_dir>/doramate_<process_id>.yml
```

如果未显式指定工作目录，则退回到当前进程的 `current_dir()`。

这比简单写入临时目录更适合当前场景，因为：

- 相对路径节点更容易解析
- 用户更容易定位运行文件
- 与打开/保存的项目目录一致

### 4. 启动前会确保 DORA runtime 就绪

在真正调用 `dora start` 前，LocalAgent 会执行：

```rust
ensure_dora_runtime_ready().await
```

这一步会：

- 检查 coordinator 是否运行
- 检查 daemon 是否运行
- 必要时尝试自动启动二者

当前配置端口为：

- `DORA_COORDINATOR_PORT = 54500`
- `DORA_CONTROL_PORT = 6012`
- `DORA_DAEMON_LOCAL_PORT = 54501`

### 5. `dora start` 已具备超时、重试和诊断

当前关键常量为：

- `DORA_START_TIMEOUT_SECS = 20`
- `DORA_START_MAX_ATTEMPTS = 2`
- `DORA_START_RETRY_DELAY_MS = 800`

当前策略是：

1. 执行一次 `dora start`
2. 最长等待 20 秒
3. 对瞬时性失败最多重试一次
4. 对明显永久错误不重试

可重试的典型情况包括：

- 无进程输出
- 超时
- connection refused
- coordinator / daemon 暂未就绪

不重试的典型情况包括：

- YAML 无效
- 未知节点
- 文件不存在

### 6. 成功后会记录两个不同 ID

当前会保存两个标识：

- `process_id`: DoraMate 内部追踪 ID
- `dataflow_uuid`: 从 `dora start` 输出中解析出的 DORA 数据流 UUID

后续停止流程时，LocalAgent 会优先使用 `dataflow_uuid` 调 `dora stop`。

### 7. 停止后会尝试清理残留节点进程

`/api/stop` 成功调用 `dora stop` 后，还会根据 YAML 中的节点 `path` 与 `id` 推断可能的进程名，并在不同平台上清理残留进程：

- Windows: `tasklist` + `taskkill`
- Unix: `pgrep` + `pkill`

这一步是对“控制平面已停但节点进程未退出”的现实补偿。

### 8. 当前 stop 语义偏宽松

即使部分停止失败，当前返回仍可能是：

- `success: true`
- `error_code: STOP_PARTIAL_FAILURE`

这代表“操作已执行，但存在告警”，而不是严格的全成功 / 全失败二元结果。

## 状态与日志监控

### 1. 当前状态不是直接读取 DORA 官方状态表

`/api/status/:process_id` 的状态快照来自本地推断，而不是直接读取某个 DORA 内部状态 API。

当前流程是：

1. 根据 `process_id` 找到 `yaml_path`
2. 重新读取 YAML
3. 解析出节点列表
4. 根据节点路径和类型推断对应进程名
5. 在本机查找这些进程是否存在

返回结果包括：

- `total_nodes`
- `running_nodes`
- `error_nodes`
- `node_details`

### 2. 节点状态识别当前是启发式的

例如：

- 路径中包含 `opencv` / `camera` 时，会推断为 `opencv-video-capture`
- 包含 `yolo` 时，会推断为 `dora-yolo`
- 包含 `rerun` / `plot` 时，会推断为 `dora-rerun`

因此当前状态监控应理解为：

> 实用的进程级观测，而不是严格的 runtime telemetry。

### 3. 状态流每 800ms 推送一次

`/api/status-stream/:process_id` 的实现是固定间隔推送：

- 周期: 800ms
- 内容: 完整 `DataflowStatusResponse`
- 当状态变为 `stopped` 或 `not_found` 时结束连接

### 4. 日志流包含 backlog 回放

每个 `DoraProcess` 都维护：

- `broadcast::Sender<LogEntry>`
- `VecDeque<LogEntry>` backlog

backlog 上限为：

```rust
const LOG_BACKLOG_LIMIT: usize = 1000;
```

新的 WebSocket 客户端接入时，LocalAgent 会先发送：

1. 一条连接成功系统消息
2. backlog 中的历史日志
3. 后续实时日志

### 5. 后端会尝试从日志中提取 `node_id`

后端当前会从日志文本中用启发式规则提取 `node_id`，用于前端做节点过滤。

因此日志面板已经具备“按节点观察日志”的基础能力。

## 文件系统与模板配置

LocalAgent 不只是执行代理，也承担浏览器无法直接完成的本地文件系统能力。

### 1. 原生目录选择

`/api/select-directory` 通过 `rfd::FileDialog::pick_folder()` 打开系统目录选择器。

前端工作目录设置功能依赖这条接口。

### 2. 原生文件打开

`/api/open-dataflow-file` 会：

1. 打开系统文件选择器
2. 限定 `yml` / `yaml`
3. 读取文件内容
4. 返回 `file_path`、`file_name`、`working_dir`

### 3. 原生文件保存

当前支持两种方式：

- `/api/save-dataflow-file`: 通过系统保存对话框决定路径
- `/api/write-dataflow-file`: 前端已知路径时直接保存

### 4. 阻塞文件操作被包装进 `spawn_blocking`

目录选择、文件选择和文件写入等同步系统调用，当前都通过 `tokio::task::spawn_blocking` 包裹，避免阻塞异步 runtime。

### 5. 节点模板配置保存在用户配置目录

当前模板配置文件位置为：

- Windows: `%APPDATA%\\DoraMate\\node_templates.yml`
- 回退: `%USERPROFILE%\\AppData\\Roaming\\DoraMate\\node_templates.yml`
- Linux/macOS: `$XDG_CONFIG_HOME/doramate/node_templates.yml`
- 回退: `~/.config/doramate/node_templates.yml`

### 6. 模板配置保存前会被标准化

当前后端会对模板配置做：

- 空白裁剪
- 默认名称补全
- 默认图标补全
- 输入输出端口去重
- 相同 `node_type` 合并覆盖

这说明 LocalAgent 不是单纯文件仓库，而是开始承担本地配置规范化职责。

## 当前实现的约束

这部分必须明确写出，避免文档再次变成“规划替代现实”。

### 1. LocalAgent 仍是单文件单体

当前 `doramate-localagent` 的核心能力都堆在 [`main.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-localagent/src/main.rs) 中。

优点：

- 开发快
- 逻辑集中

代价：

- 路由、runtime、文件系统、模板配置、测试耦合
- 后续维护和拆分成本会上升

### 2. 状态监控仍是启发式

当前状态来源依赖：

- YAML 解析
- 路径命名
- 进程名推断

所以它更适合当前阶段的“可用监控”，还不是工业级的官方运行态数据源。

### 3. 健康诊断前后端未完全打通

后端健康检查已经区分：

- `dora_installed`
- `dora_coordinator_running`
- `dora_daemon_running`

但前端目前还没有完整展示这三类状态。

### 4. 安全前提是 localhost

当前 LocalAgent 监听：

```text
127.0.0.1:52100
```

同时 CORS 设置为全开放：

```rust
allow_origin(Any)
allow_methods(Any)
allow_headers(Any)
```

这在 localhost 工具场景下可接受，但前提必须是：

- 端口不暴露到公网
- 不作为远程服务直接开放

### 5. 文档中不能再引用不存在的能力

当前仓库中并不存在这些能力，文档不应再继续描述为“已实现”：

- 安装器体系
- 系统托盘
- 自动启动服务注册
- 单独的 `runner.rs` 执行抽象
- 跨平台安装包流水线

这些内容如果后续实现，应单独补回，而不是在当前文档中继续保留。

## 总结

基于当前代码，DoraMate 的本地执行架构已经形成了完整闭环：

1. 前端完成可视化编排与 YAML 生成
2. LocalAgent 接管浏览器无法完成的系统能力
3. LocalAgent 负责确保 DORA runtime 可用
4. `dora start` / `dora stop` 完成真正的数据流执行控制
5. 前端通过状态流和日志流持续观察运行过程

因此当前 DoraMate 的本地执行能力，准确表述应为：

> 基于浏览器前端 + 本地代理 + DORA runtime 的单机执行架构。

它已经不是纯概念验证，但也还没有演进到更强的模块化和精确 runtime telemetry 阶段。当前最重要的价值是：整条链路已经真实可用，并且代码、接口和 UI 已经基本对齐。
