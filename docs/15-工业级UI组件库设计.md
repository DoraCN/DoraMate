## 定位说明

旧版本文档把 DoraMate 前端描述成了一个已经成型的“工业级通用组件库”，并进一步延伸到 ERP、MES、PLC、OA 等系统迁移规划。这个说法与当前仓库实际情况不一致。

基于当前代码，更准确的表述应该是：

> `doramate-frontend` 目前拥有一套围绕“可视化数据流编辑器”构建的前端组件体系，其中有一部分组件具备明显复用价值，但它还不是一个独立发布、独立版本化的通用 UI 组件库工程。

当前真实情况有三个特点：

1. 组件代码直接位于 `doramate-frontend/src/components/`
2. 组件主要为 DoraMate 编辑器服务，而不是面向通用后台场景
3. 组件之间已经形成较清晰的职责分工，但尚未抽离成单独 crate 或设计系统

因此，本篇文档的重点不是“未来要做什么组件库”，而是“当前已经有什么组件体系，它们是怎样协作的”。

## 组件分层

如果按当前实际用途分类，DoraMate 的组件可以分为三层。

### 1. 编辑器核心组件

这一层直接服务于数据流编排：

- `Canvas`
- `BezierConnection`
- `NodePanel`
- `PropertyPanel`
- `MinimalParameterEditor`

这是当前前端中最有业务含义的一层，也是最不适合被简单理解为“通用组件库”的部分。

### 2. 运行态观察组件

这一层服务于本地执行反馈：

- `StatusPanel`
- `LogPanel`

它们直接消费 LocalAgent 返回的运行状态和日志流，是 DoraMate 与纯静态编辑器最大的差异点之一。

### 3. 操作与弹窗组件

这一层负责操作入口、确认流程和用户配置：

- `Toolbar`
- `SaveFileDialog`
- `ConfirmDialog`
- `ShortcutSettingsDialog`
- `TemplateManagerDialog`

这部分组件是当前最接近“可复用 UI 组件”的区域，因为它们的业务耦合相对较低。

### 3. `Canvas`: 当前组件体系的核心

文件: [`doramate-frontend/src/components/canvas.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/components/canvas.rs)

`Canvas` 是当前前端中最核心、最复杂的组件。它负责：

- SVG 画布渲染
- 节点拖拽
- 画布平移
- 鼠标滚轮缩放
- 拖放新增节点
- 多选框选
- Shift 多选
- 连线创建
- 连线删除
- 节点删除
- 运行态颜色映射
- 自动布局后的视角聚焦

它的几个关键实现点值得单独说明。

#### 3.1 画布状态是局部信号，不依赖全局状态容器

组件内部自己维护：

- `dragging`
- `panning`
- `connecting`
- `mouse_pos`
- `canvas_scale`
- `canvas_offset`
- `marquee`

这说明当前前端对“高频交互状态”的处理很务实：把瞬时交互状态尽量留在局部，而不是抬到全局。

#### 3.2 运行态视觉直接映射 `NodeState`

当前节点的边框、填充、阴影、状态点颜色都会依据 `NodeState` 实时变化：

- `Idle`
- `Starting`
- `Running`
- `Stopped`
- `Error`

这让画布不是一个纯编辑视图，而是运行视图和编辑视图的混合体。

#### 3.3 连线创建不仅改 UI，还同步更新目标节点输入

当前从源节点拖到目标节点完成连线时，组件会同时：

- 向 `connections` 中追加连接
- 根据目标输入端口情况选择映射目标
- 将目标节点的 `inputs` 文本同步改成 `to_port: from/port`

这说明 `Canvas` 不是“只画线”，而是实际参与了数据模型维护。

## 状态管理与交互模式

当前组件体系虽然没有单独抽象成设计系统，但在状态组织上已经形成了一套稳定模式。

### 1. 以 `lib.rs` 为单页协调中心

当前应用主状态集中在 [`doramate-frontend/src/lib.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/lib.rs) 中，组件之间主要通过以下方式协作：

- `Signal<T>`
- `WriteSignal<T>`
- `Callback<T>`

也就是说，当前前端不是 Redux / MobX / Zustand 式的全局 store 架构，而是：

> 顶层状态集中管理，组件通过信号和回调协作。

### 2. 高频交互状态尽量局部化

像 `Canvas` 里的：

- 拖拽状态
- 平移状态
- 缩放状态
- 框选状态

都保存在组件内部，不抬升到应用根节点。

这是一个正确的取舍，因为这些状态更新频率高、生命周期短，而且只影响局部 UI。

### 3. 派生状态大量使用 `Signal::derive`

当前代码中大量使用：

```rust
Signal::derive(...)
```

来生成：

- 快捷键提示文本
- 按钮禁用态
- 运行文本
- 格式化时间
- 日志筛选结果
- 冲突检测结果

这说明当前前端已经稳定采用“原始状态 + 派生状态”的模式，而不是在模板里堆积重复计算。

### 4. 组件职责以“状态所有权”划分

一个简单但很重要的观察是：当前组件设计不是先抽象视觉，而是先明确谁拥有状态。

例如：

- `Toolbar` 不拥有业务动作，只发回调
- `Canvas` 拥有局部交互状态，但不拥有全局数据流
- `PropertyPanel` 直接编辑节点，但仍由上层持有 `nodes`
- `LogPanel` 自持日志列表和 websocket 生命周期

这是一种偏工程实践导向的组件设计方式。

## 样式组织方式

当前前端样式组织并不是独立设计系统，而是“组件同域 + 局部内联”的混合结构。

### 1. 组件同域 CSS 已经落地

当前已有多处组件与样式文件同域维护：

- `toolbar.rs` + `toolbar.css`
- `property_panel.rs` + `property_panel.css`
- `shortcut_settings.rs` + `shortcut_settings.css`
- `template_manager.rs` + `template_manager.css`

这与仓库约定“组件与样式同域维护”是一致的。

### 2. 一部分组件仍使用内联 `<style>`

例如：

- `StatusPanel`
- `ConfirmDialog`
- `SaveFileDialog`
- `LogPanel` 的局部结构样式

这说明当前样式体系仍处于“应用优先于抽象”的阶段。它的优点是：

- 开发快
- 样式依赖清晰

缺点是：

- 难以沉淀统一 tokens
- 视觉变量复用度不高

### 3. 当前视觉基调是编辑器式深色 UI

从代码可见，当前前端已经形成较稳定的界面基调：

- 深色背景
- 低饱和灰色面板
- 高亮蓝 / 绿 / 橙 / 红用于状态反馈
- 面板式布局
- 编辑器式滚动条、按钮、输入框

这说明当前不是一组零散组件，而是已经有一致的视觉语言。

## 当前约束

这部分需要明确写出，避免继续把当前前端说成“已成型的工业级通用组件库”。

### 1. 目前没有独立的组件库工程

当前仓库中并不存在：

- `doramate-ui-components` 独立 crate
- 组件发布流程
- 单独版本管理
- Storybook / 独立演示站

所以目前更准确的说法是“应用内组件体系”，不是“独立 UI 组件库产品”。

### 2. 当前缺少通用后台型基础组件

仓库里当前并没有这些旧文中提到的已实现组件：

- DataTable
- TreeView
- Chart
- Tabs
- Breadcrumb
- Sidebar
- 通用 Toast 系统

因此旧文中把这些写成“已实现”或“正在构建的工业级核心资产”，都不准确。

### 3. 当前组件更偏业务耦合，而不是纯抽象

例如：

- `Canvas` 强依赖 `Node` / `Connection`
- `PropertyPanel` 直接理解节点 ID、端口、输入引用
- `LogPanel` 理解 `LogEntry` 和 websocket 状态
- `StatusPanel` 理解运行节点统计

这并不是缺点，而是当前阶段的合理现实：先把产品做通，再考虑抽象。

### 4. 样式系统还没有沉淀为统一 design tokens

当前没有看到完整的：

- 颜色 token 系统
- 间距 token 系统
- 字体层级 token 系统
- 通用主题切换体系

所以样式复用目前仍以“复制组件 + 保持局部样式”方式为主。

### 5. 组件测试还不是独立体系

当前仓库的测试重点主要在：

- 数据模型
- YAML 转换
- API 客户端
- 状态逻辑

组件层尚未形成独立的视觉测试、交互测试或快照测试体系。

---

## 总结

基于当前 `doramate-frontend` 的真实实现，DoraMate 已经拥有一套结构清晰的前端组件体系，但这套体系的定位应当是：

> 面向可视化数据流编辑器的应用级组件集合，而不是已经独立成型的通用工业 UI 组件库。

当前这套组件体系已经具备以下价值：

1. 支撑完整的编辑器主界面
2. 支撑本地执行过程的状态与日志观察
3. 支撑快捷键、模板、确认、保存等辅助流程
4. 在应用内部形成了较稳定的状态管理与组件协作模式

当前最值得保留和继续演进的，不是“假想中的 ERP/MES 通用组件库故事”，而是已经落地的这些能力：

- 编辑器核心交互
- 运行态可观测性
- 配置型弹窗模式
- 顶层状态协调 + 局部信号状态

后续如果要真正把它演进为独立组件库，合理路径不是继续写规划文档，而是先做三件事：

1. 把编辑器强耦合数据模型从组件中解耦
2. 抽出统一的样式 tokens 和主题变量
3. 为可复用组件建立独立测试和演示体系
