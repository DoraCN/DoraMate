## 定位调整

旧版本文档把 DoraMate 直接扩展成了 “ERP / MES / OA / PLC / 生产调度 5 大系统统一迁移路线”，并且假设：

- 已经拥有可独立复用的工业级 UI 组件库
- 已经验证了一套稳定的通用 Rust 全栈企业架构
- 已经可以量化多个外部系统的 ROI 和工期

这些前提与当前仓库现状不一致。

基于当前代码，更准确的判断应当是：

> DoraMate 目前已经沉淀出一套围绕“可视化数据流编辑 + LocalAgent 本地执行 + 运行观测”展开的应用级资产，这些资产对某些相邻系统有明显迁移价值，但还不支持直接把任意工业业务系统整体迁移到现有架构上。

因此，本篇文档不再讨论脱离代码现实的“五大系统总体改造工程”，而是聚焦三个问题：

1. 当前仓库到底已经沉淀了哪些可迁移资产
2. 哪些系统类型和这些资产天然相邻
3. 在尝试更大范围复用前，还必须先补齐什么

## 当前项目已沉淀的资产

截至当前仓库状态，DoraMate 已经沉淀出三类真正可迁移的资产。

### 1. 前端交互资产

来自 `doramate-frontend` 的真实可迁移能力包括：

- 节点式可视化编辑器主界面
- 画布缩放、平移、框选、多选、复制粘贴、重复
- 自动布局算法
- 属性编辑面板
- YAML 读写与布局 sidecar 支持
- 运行状态面板
- 日志观察面板
- 快捷键配置 UI
- 模板管理 UI

这些能力已经在 [`doramate-frontend/src/lib.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/lib.rs) 和 [`doramate-frontend/src/components/`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-frontend/src/components/mod.rs) 中形成完整闭环。

### 2. 本地代理资产

来自 `doramate-localagent` 的真实可迁移能力包括：

- 本地 HTTP API 基础框架
- WebSocket 状态流 / 日志流
- 本地文件系统访问
- 原生目录选择 / 文件打开 / 文件保存
- 本地运行时自启动与诊断
- 外部 CLI 执行包装
- 错误码体系
- 日志回放 backlog

这些能力集中在 [`doramate-localagent/src/main.rs`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-localagent/src/main.rs) 和 [`doramate-localagent/README.md`](/c:/Users/Administrator/projects/DoraMateStudy/doramate-localagent/README.md) 中。

### 3. 工程与验证资产

当前仓库并不只有功能代码，也已经有一部分“迁移前提资产”：

- `cargo test --locked` 的本地基线
- frontend wasm target check 基线
- smoke 脚本
- 运行 / 停止 / 状态 / 文件流的关键路径测试
- Delivery status 与 sprint 文档

这类资产虽然不是产品功能，但对后续任何迁移尝试都非常重要，因为它们决定了“这套东西是否可被重复交付”。

## 哪些系统适合优先迁移

不是所有系统都应该从 DoraMate 现有仓库开始迁移。真正适合优先复用的，是那些在交互模型和运行边界上与 DoraMate 接近的系统。

### 1. 节点式编排或流程式编辑系统

最适合优先迁移的，是具备以下特征的系统：

- 有可视化节点或流程图编辑需求
- 需要连线、拓扑、依赖关系表达
- 有本地运行、调试、状态观测需求
- 可以接受“浏览器前端 + 本地代理 + 外部 runtime”的结构

典型例子：

- 本地 AI 工作流编排器
- 机器人 / 视觉 / 音频处理流程编排工具
- 节点式脚本运行器
- 本地自动化任务管线编辑器

这类系统与 DoraMate 的前端交互模型和 LocalAgent 边界高度相似，复用成本最低。

### 2. 本地工具型控制台应用

第二类适合对象，是具有明显本地系统接入需求的工具应用：

- 需要本地文件系统访问
- 需要调用本机外部命令
- 需要原生目录或文件选择框
- 需要本地状态流和日志流

典型例子：

- 本地模型运行控制台
- 多进程工具链编排器
- 设备脚本部署助手
- 本地诊断与采集工具

这类系统不一定需要 `Canvas`，但会明显受益于 `LocalAgent` 的接口与错误诊断模式。

### 3. 可视化配置编辑器

第三类适合对象，是以配置编辑和结构化文件生成为主的系统：

- 图形化编辑配置
- 导出 YAML / JSON / sidecar
- 需要模板体系
- 需要属性面板与字段校验

典型例子：

- 规则编排器
- 节点模板编辑器
- 设备配置编辑器
- 任务描述文件生成器

它们可以复用：

- `PropertyPanel`
- `MinimalParameterEditor`
- `TemplateManagerDialog`
- `SaveFileDialog`
- LocalAgent 文件接口

## 当前不适合直接迁移的系统类型

这部分比“可以做什么”更重要，因为它决定了当前技术边界。

### 1. 通用企业后台系统

当前仓库并不适合直接拿去迁移典型的：

- ERP
- OA
- CRM
- 通用 MIS

原因很直接：

- 没有 DataTable、TreeView、Tabs、权限中心等通用后台核心组件
- 没有多用户后端架构
- 没有认证、授权、角色模型
- 没有数据库抽象与迁移机制
- 当前状态模型仍然是单机单用户应用

如果强行套用，会出现“前端像编辑器，后端像本地工具，业务却是企业信息系统”的结构错位。

### 2. 需要集中式服务治理的平台系统

当前仓库也不适合直接作为以下系统的核心底座：

- 多租户 SaaS 平台
- 集中调度平台
- 云端任务中心
- 跨组织协作系统

因为当前架构的核心前提是：

- LocalAgent 运行在用户本机
- 服务监听 `127.0.0.1:52100`
- CORS 开放但默认只用于本地回环

这套设计天然偏向单机本地工具，而不是中心化在线平台。

### 3. 重数据表 / 重报表场景

当前也不适合优先迁移那些高度依赖：

- 大表格
- 复杂筛选
- 多维报表
- 树形组织结构
- 甘特图 / 时间轴

的系统。

原因不是 Rust 或 Leptos 不行，而是当前仓库里并没有对应的成熟前端资产。当前真正成熟的是编辑器和本地运行观察，而不是企业报表工作台。

## 推荐迁移路线

基于当前仓库状态，推荐的迁移路线不应该是“一次性规划 5 大系统”，而应该是从近到远逐层扩展。

### 路线 A: DoraMate 内部平台化

第一步不是迁移外部系统，而是先把 DoraMate 自己沉淀成更清晰的可复用平台。

当前最值得抽取的平台化方向有：

- 前端编辑器壳层
- 画布与节点交互层
- LocalAgent 文件与运行桥接层
- 日志与状态观察层
- 模板存储与模板管理层

如果这一步不做，后续迁移只是复制代码，而不是复用架构。

### 路线 B: 迁移到相邻的本地流程工具

在平台化初步完成后，优先迁移到“相邻产品”而不是“大型业务系统”：

- 本地节点编排工具
- 规则流配置器
- 模型推理工作流助手
- 设备任务脚本编排器

这些系统与 DoraMate 的共性更强，复用成功概率更高。

### 路线 C: 再考虑业务系统中的局部子系统

只有在前两步验证稳定后，才考虑把其中一部分能力迁移到更传统的业务系统中，但也应该是局部切入：

- 配置编辑子系统
- 流程设计器子系统
- 设备脚本配置器
- 调试控制台子系统

而不是整个 ERP / MES 一次性替换。

## 阶段性实施计划

下面这份计划基于当前交付状态、`DELIVERY_STATUS.md` 和 `NEXT_SPRINT_2026-03.md` 的内容整理，不引入脱离仓库的新假设。

### Phase 0: 完成 DoraMate 当前 MVP 收口

这是所有迁移动作的前提。当前仍在收尾的关键项包括：

- 严格 smoke 在真实 DORA 环境中通过
- LocalAgent 稳定性继续加固
- 自定义节点模板管理 UI 完整收口
- 关键用户路径 E2E 回归门禁

如果这些还没完成，就谈不上“迁移到其他系统”。

### Phase 1: 识别并抽离可复用模块

完成 MVP 收口后，优先做代码层抽离，而不是做市场层扩张。

建议优先拆分的模块：

- `frontend-editor-core`
  - 画布
  - 节点模板拖放
  - 选中态与布局逻辑
- `frontend-runtime-observability`
  - `StatusPanel`
  - `LogPanel`
  - WebSocket 客户端适配
- `localagent-core`
  - 文件读写
  - 原生对话框
  - 运行桥接
  - 错误码和日志骨架

这一步的目标不是发布新产品，而是降低下一次复用时的复制粘贴成本。

### Phase 2: 做一个相邻系统试点

抽离后，选择一个与 DoraMate 最接近的试点系统，范围应控制在：

- 单机
- 本地运行
- 流程式配置
- 用户数量少
- 数据量中等

试点目标不是“大而全上线”，而是验证三个问题：

1. 当前交互资产能否在新场景中保持可用
2. LocalAgent 模式是否足够通用
3. 编辑器数据模型是否需要继续解耦

### Phase 3: 扩展到局部业务子系统

只有在试点成功后，才适合把能力复用到更传统的业务系统中，但仍建议以“内嵌工具子系统”形式切入：

- 设备配置器
- 调试面板
- 流程设计器
- 规则编辑器

这一步的价值是：把 DoraMate 的资产当作“业务系统中的专业编辑器模块”，而不是整个平台底座。

### Phase 4: 再评估是否值得建设真正的通用平台

到这一步，才有资格讨论：

- 是否抽离成独立组件库
- 是否抽离成独立 LocalAgent SDK
- 是否统一到新的组织级前端 / 本地工具框架

在此之前，过早谈“统一五大系统迁移”只会制造幻觉。

## 主要风险

### 1. 把应用级代码误判为平台级资产

当前很多代码虽然可复用，但仍强耦合于 DoraMate 语义，例如：

- `Node`
- `Connection`
- DORA YAML
- `NodeState`
- 本地运行 / 停止模型

如果不先拆语义边界，迁移时会发现“表面通用，实际到处改”。

### 2. 过早迁移到不相邻系统

当前最危险的路线，是直接从 DoraMate 跳到：

- ERP
- MES 主系统
- OA 主系统

因为这些系统真正需要的核心资产，当前仓库并没有：

- 表格体系
- 权限体系
- 业务数据模型
- 数据库访问层
- 多用户后台治理

### 3. LocalAgent 模式不适合所有部署场景

LocalAgent 当前的价值在于本地执行与系统接入，但它的前提也是约束：

- 依赖用户本机安装
- 依赖本地 CLI
- 依赖 localhost 通信

这套模式不适合直接搬到需要纯浏览器 / 纯云端的系统中。

### 4. 当前验证基线仍偏 MVP

虽然当前已经有：

- 单元测试
- wasm target check
- smoke 脚本

但离“可作为跨产品平台交付”仍有差距，尤其是：

- 真环境严格 smoke
- E2E 回归门禁
- 更细的模块边界测试

---

## 总结

基于当前 `doramate-frontend` 和 `doramate-localagent` 的真实实现，DoraMate 的迁移规划应该从“宏大承诺”调整为“现实复用路线”。

最准确的结论是：

> DoraMate 当前已经沉淀出一套对本地流程编排工具高度有价值的前端与本地代理资产，但它还不是可以直接承接通用企业系统整体迁移的平台。

当前真正值得执行的迁移策略是：

1. 先完成 DoraMate 当前 MVP 收口
2. 抽离编辑器、本地代理、运行观测等可复用模块
3. 在相邻的本地流程类工具中做试点复用
4. 只在验证充分后，再考虑向更传统的业务系统扩展

这条路线虽然没有旧版文档里那么“宏伟”，但它与当前代码现状一致，也更有成功概率。
